# 임베디드 SW 종류

## 1. 수행 목표

본 문서는 임베디드 시스템에서 사용되는 소프트웨어의 종류를 이해하기 위해 작성하였다.
주요 내용은 펌웨어의 정의, RTOS의 정의, 펌웨어와 RTOS의 차이, 우선순위, 우선순위 역전, DeadLock, 스케줄러의 역할이다.

---

## 2. 펌웨어의 정의

펌웨어(Firmware)는 하드웨어 장치 안에 저장되어 해당 장치를 제어하는 소프트웨어이다.
일반적인 응용 프로그램처럼 사용자가 자유롭게 실행하고 종료하는 프로그램이 아니라, 하드웨어가 정상적으로 동작하기 위해 장치 내부에서 실행되는 제어 코드이다.

예를 들어 다음과 같은 장치에는 펌웨어가 들어 있다.

| 장치      | 펌웨어 역할                   |
| ------- | ------------------------ |
| 전자레인지   | 버튼 입력 처리, 타이머 제어, 출력 제어  |
| 공유기     | 네트워크 연결, IP 할당, 무선 통신 제어 |
| 프린터     | 인쇄 명령 처리, 모터 제어, 용지 감지   |
| 스마트폰    | 부팅, 통신 모듈 제어, 카메라 모듈 제어  |
| 자동차 ECU | 센서 입력 처리, 엔진/모터/브레이크 제어  |
| 아두이노    | 센서값 읽기, LED/모터/릴레이 제어    |

펌웨어는 하드웨어와 소프트웨어의 중간에 있는 개념처럼 보이지만, 실제로는 **하드웨어를 직접 제어하기 위해 장치 내부에 저장된 소프트웨어**라고 이해하면 된다.

---

## 3. 펌웨어가 저장되는 장치

펌웨어는 일반적으로 전원이 꺼져도 내용이 지워지지 않는 비휘발성 메모리에 저장된다.

| 저장 장치        | 특징                             |
| ------------ | ------------------------------ |
| ROM          | 제조 시 기록되며 수정이 어렵다.             |
| Flash Memory | 전원이 꺼져도 데이터가 유지되고 업데이트가 가능하다.  |
| EEPROM       | 소량의 설정값 저장에 사용된다.              |
| MCU 내부 Flash | 아두이노, STM32 같은 마이크로컨트롤러에 내장된다. |

펌웨어를 비휘발성 메모리에 저장하는 이유는 전원이 꺼졌다 켜져도 장치가 다시 동작해야 하기 때문이다.
예를 들어 아두이노에 업로드한 프로그램은 전원을 껐다 켜도 다시 실행된다. 이는 펌웨어가 플래시 메모리에 저장되어 있기 때문이다.

---

## 4. 스마트폰에서 펌웨어와 App의 차이

스마트폰에는 펌웨어와 일반 App이 모두 존재한다.

| 구분    | 펌웨어                            | App                        |
| ----- | ------------------------------ | -------------------------- |
| 위치    | 시스템 내부, 하드웨어 제어 영역             | 사용자 영역                     |
| 역할    | 부팅, 통신, 카메라, 센서, 배터리 등 하드웨어 제어 | 메신저, 게임, 웹브라우저 등 사용자 기능 제공 |
| 변경 방식 | 제조사 업데이트 또는 시스템 업데이트           | 사용자가 설치/삭제 가능              |
| 실행 권한 | 높은 권한 필요                       | 제한된 권한에서 실행                |
| 오류 영향 | 장치 부팅 불량, 통신 불량 등 심각한 문제 가능    | 특정 앱만 종료되거나 오류 발생          |

예를 들어 카카오톡, 유튜브, 웹브라우저는 App이다.
반면 스마트폰의 통신 모뎀 제어 코드, 부트로더, 카메라 모듈 제어 코드, 배터리 관리 코드는 펌웨어에 가깝다.

## 5. 펌웨어 업데이트 방법

펌웨어는 장치 기능 개선, 오류 수정, 보안 패치, 새로운 하드웨어 호환을 위해 업데이트된다.

| 업데이트 방법     | 설명                          |
| ----------- | --------------------------- |
| USB 연결 업데이트 | PC와 장치를 USB로 연결하여 펌웨어를 업로드  |
| OTA 업데이트    | Wi-Fi나 통신망을 통해 무선으로 업데이트    |
| 전용 프로그래머 사용 | ISP, JTAG, SWD 같은 장비로 직접 기록 |
| SD 카드 업데이트  | 펌웨어 파일을 SD 카드에 넣고 장치에서 업데이트 |
| 제조사 업데이트 도구 | 제조사가 제공하는 프로그램으로 업데이트       |

아두이노의 경우 PC에서 Arduino IDE로 작성한 코드를 USB로 업로드하면 부트로더가 새 펌웨어를 플래시 메모리에 기록한다.

## 6. RTOS의 정의

RTOS(Real-Time Operating System)는 실시간 운영체제이다.
정해진 시간 안에 작업을 처리해야 하는 시스템을 위해 설계된 운영체제이다.

일반 운영체제는 평균적인 처리 성능과 사용자 편의성이 중요하다.
반면 RTOS는 특정 작업이 **언제 실행되고, 언제 끝나는지 예측 가능해야 한다.**

예를 들어 다음 시스템에서는 실시간성이 중요하다.

| 시스템     | 실시간성이 필요한 이유               |
| ------- | -------------------------- |
| 자동차 에어백 | 충돌 감지 후 매우 짧은 시간 안에 동작해야 함 |
| 로봇 제어기  | 제어 주기마다 모터 명령을 정확히 출력해야 함  |
| 의료기기    | 환자 상태 변화에 정해진 시간 안에 반응해야 함 |
| 드론 제어기  | 자세 제어가 늦어지면 추락할 수 있음       |
| 산업용 장비  | 센서 입력과 모터 출력 타이밍이 중요함      |

---

## 7. Real-time의 의미

Real-time은 단순히 “빠르다”는 뜻이 아니다.
정확한 의미는 **정해진 시간 제한 안에 반드시 처리해야 한다**는 것이다.

예를 들어 로봇 제어 시스템이 1ms 주기로 동작한다면, 센서 읽기, 제어 계산, 모터 출력이 1ms 안에 끝나야 한다.
평균적으로 빠른 것보다 최악의 경우에도 정해진 시간 안에 끝나는 것이 중요하다.

```text
센서 입력 → 제어 연산 → 모터 출력
          1ms 안에 완료
```

## 8. 펌웨어와 RTOS의 차이

| 구분    | 펌웨어                                   | RTOS                                           |
| ----- | ------------------------------------- | ---------------------------------------------- |
| 정의    | 하드웨어를 직접 제어하는 소프트웨어                   | 실시간 작업 관리를 위한 운영체제                             |
| 구조    | 보통 단일 반복문 또는 interrupt 중심             | task, scheduler, priority, semaphore, mutex 사용 |
| 실행 방식 | 정해진 코드가 순차적으로 실행                      | 여러 task를 우선순위에 따라 실행                           |
| 멀티태스킹 | 직접 구현해야 함                             | RTOS가 task 전환을 관리                              |
| 실시간성  | 단순한 실시간 제어 가능                         | 복잡한 실시간 제어에 적합                                 |
| 예시    | Arduino loop(), STM32 bare-metal code | FreeRTOS, VxWorks, RTEMS, Zephyr               |

펌웨어는 하드웨어 제어 코드 자체이고, RTOS는 여러 펌웨어 기능 또는 task를 시간 조건에 맞게 관리하는 운영체제이다.

---

## 9. 펌웨어로 구현하기 어려운 기능

간단한 장치 제어는 펌웨어만으로 충분하다.
하지만 기능이 많아지고 동시에 처리해야 할 작업이 늘어나면 펌웨어만으로 구현하기 어려워진다.

| 구현하기 어려운 기능   | 이유                                    |
| ------------- | ------------------------------------- |
| 복잡한 멀티태스킹     | 여러 작업의 실행 순서를 직접 관리해야 함               |
| 우선순위 기반 작업 처리 | 어떤 작업을 먼저 실행할지 판단하는 구조가 복잡함           |
| 정교한 시간 관리     | 주기적 task 실행과 지연 시간 관리가 어려움            |
| 공유 자원 관리      | 여러 작업이 같은 자원을 사용할 때 충돌 가능             |
| 네트워크 스택       | TCP/IP, Bluetooth 등 복잡한 통신 구조 구현이 어려움 |
| 파일 시스템        | 저장장치 관리와 데이터 구조가 복잡함                  |
| GUI 처리        | 화면 출력, 입력 이벤트 처리 등 많은 기능 필요           |

따라서 단순 센서 제어와 LED 제어는 펌웨어로 가능하지만, 복잡한 로봇 제어, 통신, 로그 저장, 여러 task 동시 실행은 RTOS나 임베디드 리눅스가 더 적합하다.

---

## 10. 펌웨어에서 멀티태스킹이 어려운 이유

일반적인 펌웨어는 다음과 같은 구조를 가진다.

```c
void setup() {
    // 초기 설정
}

void loop() {
    // 반복 실행
}
```

이 구조는 이해하기 쉽지만, 여러 작업을 동시에 처리하기 어렵다.

예를 들어 다음 작업을 동시에 처리해야 한다고 가정한다.

```text
1. 센서값 읽기
2. 모터 제어
3. 통신 데이터 수신
4. LED 상태 표시
5. 에러 감시
```

단순 loop 구조에서는 한 작업이 오래 걸리면 다른 작업이 지연된다.
따라서 여러 작업의 우선순위와 실행 주기를 정밀하게 관리하려면 RTOS가 필요하다.

---

## 11. 프로그램 간 우선순위가 필요한 이유

임베디드 시스템에서는 모든 작업이 같은 중요도를 가지지 않는다.
어떤 작업은 늦어져도 큰 문제가 없지만, 어떤 작업은 반드시 먼저 처리되어야 한다.

예를 들어 로봇 제어 시스템에서 다음 작업들이 있다고 가정한다.

| 작업          | 중요도   |
| ----------- | ----- |
| 비상 정지 입력 처리 | 매우 높음 |
| 모터 제어 주기 실행 | 높음    |
| 센서 데이터 수집   | 중간    |
| LCD 화면 출력   | 낮음    |
| 로그 저장       | 낮음    |

비상 정지 입력은 화면 출력보다 먼저 처리되어야 한다.
따라서 RTOS에서는 task마다 우선순위를 지정하고, 중요한 작업이 먼저 CPU를 사용할 수 있도록 한다.

---

## 12. 우선순위의 개념과 용도

우선순위는 RTOS 스케줄러가 어떤 task를 먼저 실행할지 결정하는 기준이다.

```text
높은 우선순위 task → 먼저 실행
낮은 우선순위 task → CPU가 남을 때 실행
```

우선순위는 다음과 같은 용도로 사용된다.

1. 실시간성이 중요한 작업을 먼저 실행한다.
2. 안전 관련 작업을 지연 없이 처리한다.
3. 주기적 제어 task의 실행 시간을 보장한다.
4. 중요하지 않은 로그 저장, 화면 출력은 낮은 우선순위로 실행한다.

---

## 13. 우선순위 역전 현상

우선순위 역전(Priority Inversion)은 높은 우선순위 task가 낮은 우선순위 task 때문에 실행되지 못하는 상황을 말한다.

### 13.1 예시 상황

```text
1. 낮은 우선순위 task L이 공유 자원 A를 사용하기 위해 mutex를 잠근다.
2. 높은 우선순위 task H가 같은 공유 자원 A를 사용하려고 한다.
3. 하지만 A는 L이 사용 중이므로 H는 대기한다.
4. 이때 중간 우선순위 task M이 실행되면 L은 CPU를 받지 못한다.
5. 결과적으로 H는 M과 L 때문에 계속 기다리게 된다.
```

이 상황에서는 원래 가장 먼저 실행되어야 할 높은 우선순위 task H가 실제로는 늦게 실행된다.
이것이 우선순위 역전이다.

### 13.2 해결 방법

| 해결 방법                | 설명                                           |
| -------------------- | -------------------------------------------- |
| Priority Inheritance | 낮은 우선순위 task가 mutex를 잡고 있을 때 임시로 높은 우선순위를 부여 |
| Priority Ceiling     | 공유 자원을 사용할 때 미리 정해진 높은 우선순위로 실행              |
| Critical Section 최소화 | mutex를 잡고 있는 시간을 최대한 짧게 유지                   |
| 공유 자원 줄이기            | 여러 task가 같은 자원에 접근하는 구조를 줄임                  |

가장 대표적인 해결 방법은 **Priority Inheritance**이다.

## 14. DeadLock의 정의

DeadLock은 여러 task 또는 프로그램이 서로가 가진 자원을 기다리면서 영원히 진행하지 못하는 상태를 말한다.

예를 들어 두 task가 있다고 가정한다.

```text
Task A: 자원 1을 잡고 자원 2를 기다림
Task B: 자원 2를 잡고 자원 1을 기다림
```

이 경우 Task A와 Task B는 서로가 자원을 놓기만 기다리므로 둘 다 실행을 계속할 수 없다.
이 상태가 DeadLock이다.

---

## 15. DeadLock이 일어나는 원인

DeadLock은 보통 다음 조건들이 동시에 만족될 때 발생한다.

| 조건     | 설명                              |
| ------ | ------------------------------- |
| 상호 배제  | 하나의 자원을 한 번에 하나의 task만 사용할 수 있음 |
| 점유와 대기 | task가 자원을 가진 상태에서 다른 자원을 기다림    |
| 비선점    | 다른 task가 가진 자원을 강제로 빼앗을 수 없음    |
| 순환 대기  | task들이 서로가 가진 자원을 원형으로 기다림      |

이 네 가지 조건이 동시에 만족되면 DeadLock이 발생할 가능성이 높다.

---

## 16. DeadLock의 가장 쉬운 해결 방법

가장 쉬운 해결 방법은 **자원을 항상 같은 순서로 요청하게 만드는 것**이다.

예를 들어 모든 task가 다음 순서로만 자원을 요청하도록 정한다.

```text
항상 자원 A를 먼저 요청
그 다음 자원 B를 요청
그 다음 자원 C를 요청
```

이렇게 하면 순환 대기가 발생하기 어렵다.

### 예시

잘못된 방식:

```text
Task 1: Mutex A → Mutex B 요청
Task 2: Mutex B → Mutex A 요청
```

개선된 방식:

```text
Task 1: Mutex A → Mutex B 요청
Task 2: Mutex A → Mutex B 요청
```

두 task가 같은 순서로 자원을 요청하면 DeadLock 가능성이 줄어든다.

---

## 17. DeadLock 해결 방법 정리

| 해결 방법               | 설명                        |
| ------------------- | ------------------------- |
| 자원 요청 순서 고정         | 모든 task가 같은 순서로 mutex를 요청 |
| Timeout 사용          | 일정 시간 안에 자원을 얻지 못하면 포기    |
| Lock 사용 시간 최소화      | mutex를 잡고 있는 시간을 줄임       |
| Nested Lock 피하기     | 여러 mutex를 동시에 잡는 구조를 줄임   |
| DeadLock 감시 task 사용 | 일정 시간 응답이 없으면 오류로 처리      |
| Watchdog Timer 사용   | 시스템이 멈추면 재시작              |

임베디드 시스템에서는 DeadLock이 발생하면 장치가 멈출 수 있으므로 설계 단계에서 예방하는 것이 가장 중요하다.

---

## 18. 스케줄러의 정의

스케줄러(Scheduler)는 여러 task 중에서 어떤 task를 언제 실행할지 결정하는 시스템 프로그램이다.
RTOS에서 스케줄러는 CPU 사용 순서를 결정하는 핵심 기능이다.

스케줄러는 다음과 같은 정보를 기준으로 task 실행을 결정한다.

| 기준      | 설명                            |
| ------- | ----------------------------- |
| 우선순위    | 높은 우선순위 task를 먼저 실행           |
| task 상태 | Ready, Running, Blocked 상태 확인 |
| 시간      | 주기적 task 실행 시간 확인             |
| 이벤트     | 인터럽트, 세마포어, 메시지 큐 등 확인        |

---

## 19. 스케줄러가 쓰이는 용도

스케줄러는 다음과 같은 용도로 사용된다.

1. 여러 task를 번갈아 실행한다.
2. 높은 우선순위 task를 먼저 실행한다.
3. 주기적인 제어 task를 정해진 시간에 실행한다.
4. 통신, 센서, 제어, 로그 task를 분리하여 관리한다.
5. 인터럽트 이후 실행할 task를 결정한다.
6. CPU 시간을 효율적으로 분배한다.

예를 들어 로봇 제어기에서는 다음과 같이 task를 나눌 수 있다.

| Task               | 역할       | 우선순위  |
| ------------------ | -------- | ----- |
| Emergency Task     | 비상 정지 확인 | 매우 높음 |
| Motor Control Task | 모터 제어    | 높음    |
| Sensor Task        | 센서값 읽기   | 중간    |
| Communication Task | 외부 통신    | 중간    |
| Log Task           | 데이터 저장   | 낮음    |
| Display Task       | 화면 출력    | 낮음    |

---

## 20. 시스템 프로그램과 일반 프로그램의 차이

| 구분      | 시스템 프로그램                 | 일반 프로그램           |
| ------- | ------------------------ | ----------------- |
| 역할      | 운영체제, 스케줄러, 드라이버, 메모리 관리 | 사용자가 원하는 기능 수행    |
| 권한      | 상대적으로 높은 권한              | 제한된 권한            |
| 하드웨어 접근 | 직접 접근 가능                 | 보통 시스템 호출을 통해 접근  |
| 예시      | 스케줄러, 커널, 디바이스 드라이버      | 메신저, 웹브라우저, 계산기 앱 |
| 오류 영향   | 전체 시스템에 영향 가능            | 해당 프로그램만 종료 가능    |

RTOS에서는 일반 PC 운영체제보다 시스템 프로그램과 응용 프로그램의 경계가 명확하지 않을 수 있다.
그 이유는 작은 임베디드 시스템에서는 하나의 펌웨어 안에 드라이버, 제어 로직, 통신 로직, 응용 기능이 함께 들어가는 경우가 많기 때문이다.

---

## 21. RTOS에서 응용 프로그램과 시스템 프로그램의 경계가 명확하지 않은 이유

일반 PC에서는 운영체제 커널과 응용 프로그램이 명확히 구분된다.
하지만 RTOS 기반 임베디드 시스템에서는 다음 이유로 경계가 흐릴 수 있다.

1. 시스템 규모가 작다.
2. 하나의 펌웨어 이미지 안에 RTOS와 사용자 코드가 함께 빌드된다.
3. task가 직접 하드웨어 드라이버를 호출하는 경우가 많다.
4. 메모리 보호 기능이 없는 MCU도 많다.
5. 성능과 실시간성을 위해 추상화 계층을 줄이는 경우가 있다.

따라서 RTOS에서는 응용 코드가 시스템 자원에 가까이 접근하는 경우가 많다.

---

## 22. RTOS 디버깅 및 문제 해결

RTOS 기반 시스템에서는 단순한 코드 오류뿐 아니라 타이밍 문제도 중요하다.

| 문제                 | 설명                                | 해결 방법                   |
| ------------------ | --------------------------------- | ----------------------- |
| DeadLock           | task들이 서로 자원을 기다리며 멈춤             | 자원 순서 고정, timeout 사용    |
| Priority Inversion | 높은 우선순위 task가 낮은 우선순위 task 때문에 지연 | Priority Inheritance 사용 |
| Stack Overflow     | task stack 크기 부족                  | stack 사용량 확인 및 크기 조정    |
| Race Condition     | 여러 task가 공유 자원을 동시에 접근            | mutex, semaphore 사용     |
| Timing Miss        | 정해진 시간 안에 작업을 못 끝냄                | 코드 최적화, 우선순위 조정         |
| Starvation         | 낮은 우선순위 task가 계속 실행되지 못함          | 우선순위 재조정, aging 적용      |

RTOS 디버깅에서는 단순히 결과값만 보는 것이 아니라, task 실행 순서, 실행 시간, 대기 시간, mutex 점유 상태를 함께 확인해야 한다.

---

## 23. 칩셋에 따른 소프트웨어 선택

칩셋 성능과 메모리 용량에 따라 사용할 수 있는 소프트웨어가 달라진다.

| 칩셋 종류      | 사용 가능한 소프트웨어             | 특징                      |
| ---------- | ------------------------ | ----------------------- |
| 8-bit MCU  | Bare-metal 펌웨어, 간단한 RTOS | 메모리와 성능이 제한적            |
| 16-bit MCU | 펌웨어, 소형 RTOS             | 간단한 제어와 통신 가능           |
| 32-bit MCU | 펌웨어, RTOS                | 복잡한 제어, 통신, 멀티태스킹 가능    |
| 64-bit MPU | 부트로더, 펌웨어, 임베디드 리눅스      | 고성능 연산, 네트워크, 파일 시스템 가능 |

---

## 24. 8-bit 칩셋에 RTOS 탑재가 가능한가

8-bit 칩셋에도 RTOS를 탑재할 수는 있다.
예를 들어 작은 RTOS 커널을 사용하면 task 전환, delay, semaphore 같은 기능을 구현할 수 있다.

하지만 다음과 같은 제한이 있다.

| 제한 사항    | 설명                        |
| -------- | ------------------------- |
| RAM 부족   | 여러 task의 stack을 저장하기 어렵다. |
| 처리 속도 제한 | context switching 부담이 크다. |
| 주변장치 제한  | 복잡한 통신 기능을 동시에 처리하기 어렵다.  |
| 디버깅 어려움  | 작은 시스템에서 원인 분석이 어렵다.      |

따라서 8-bit MCU에서는 RTOS를 사용할 수는 있지만, 단순한 펌웨어 구조가 더 적합한 경우가 많다.

---

## 25. RTOS 실행의 최소 조건

RTOS가 실행되려면 최소한 다음 요소가 필요하다.

| 조건                   | 설명                           |
| -------------------- | ---------------------------- |
| CPU                  | task를 실행할 연산 장치              |
| RAM                  | 각 task의 stack과 RTOS 관리 정보 저장 |
| Flash/ROM            | RTOS 코드와 사용자 프로그램 저장         |
| Timer                | 일정 주기로 scheduler tick 발생     |
| Interrupt            | 외부 이벤트와 시간 이벤트 처리            |
| Context Switching 기능 | task 상태 저장 및 복원              |
| C Compiler/Toolchain | RTOS와 응용 코드 빌드               |

특히 timer interrupt와 context switching 기능은 RTOS 동작에 매우 중요하다.

---

## 26. 32-bit와 64-bit에서도 펌웨어 사용이 가능한가

펌웨어는 8-bit MCU에서만 사용하는 개념이 아니다.
32-bit MCU와 64-bit 프로세서에서도 펌웨어는 사용된다.

| 시스템         | 펌웨어 사용 예                          |
| ----------- | --------------------------------- |
| 8-bit MCU   | Arduino sketch, 센서 제어 코드          |
| 32-bit MCU  | STM32 펌웨어, FreeRTOS 기반 제어 코드      |
| 64-bit 프로세서 | BIOS, UEFI, Bootloader, 장치 초기화 코드 |

다만 64-bit 프로세서에서는 펌웨어가 보통 운영체제를 실행하기 전 하드웨어 초기화와 부팅을 담당한다.
반면 8-bit 또는 32-bit MCU에서는 펌웨어 자체가 장치의 전체 기능을 수행하는 경우가 많다.

---

## 27. 펌웨어나 임베디드 리눅스보다 RTOS가 필요한 경우

펌웨어는 단순 제어에 적합하고, 임베디드 리눅스는 복잡한 기능과 네트워크, 파일 시스템에 적합하다.
하지만 일정한 시간 안에 반드시 처리해야 하는 제어 시스템에서는 RTOS가 필요하다.

RTOS만의 특징은 다음과 같다.

| 특징                   | 설명                         |
| -------------------- | -------------------------- |
| 예측 가능한 응답 시간         | task가 언제 실행될지 예측 가능        |
| 우선순위 기반 스케줄링         | 중요한 task를 먼저 실행            |
| 낮은 interrupt latency | 외부 이벤트에 빠르게 반응             |
| task 간 동기화           | mutex, semaphore, queue 제공 |
| 실시간 제어 적합            | 모터, 로봇, 자동차 제어에 적합         |

---

## 28. 임베디드 리눅스가 점점 많이 쓰이는 이유

임베디드 리눅스는 펌웨어나 RTOS보다 무겁지만, 다음과 같은 장점 때문에 많이 사용된다.

| 이유       | 설명                                   |
| -------- | ------------------------------------ |
| 네트워크 기능  | TCP/IP, Wi-Fi, Bluetooth 등을 쉽게 사용 가능 |
| 파일 시스템   | 로그 저장, 설정 파일 관리가 쉬움                  |
| 드라이버 지원  | 다양한 하드웨어 드라이버 사용 가능                  |
| 개발 편의성   | Python, C/C++, Shell 등 다양한 개발 가능     |
| GUI 지원   | 화면 출력과 사용자 인터페이스 구현 가능               |
| 오픈소스 생태계 | 라이브러리와 도구가 풍부함                       |

따라서 복잡한 IoT 장비, 로봇, 스마트 기기, 네트워크 장비에서는 임베디드 리눅스가 자주 사용된다.
하지만 매우 짧은 응답 시간이 필요한 hard real-time 제어에서는 RTOS가 더 적합할 수 있다.

## 29. 전체 비교 정리

| 구분      | 펌웨어               | RTOS             | 임베디드 리눅스                      |
| ------- | ----------------- | ---------------- | ----------------------------- |
| 목적      | 하드웨어 직접 제어        | 실시간 task 관리      | 복잡한 응용 프로그램 실행                |
| 구조      | 단순 loop/interrupt | scheduler + task | kernel + process + filesystem |
| 실시간성    | 단순 제어 가능          | 높음               | 일반적으로 낮음                      |
| 메모리 요구량 | 작음                | 중간               | 큼                             |
| 개발 난이도  | 낮음                | 중간               | 높음                            |
| 멀티태스킹   | 직접 구현             | RTOS가 지원         | OS가 지원                        |
| 사용 예    | 아두이노, 센서 제어       | 로봇 제어기, 자동차 ECU  | 라즈베리파이, 스마트 장비                |

## 30. 전체 흐름 정리

```text
펌웨어
→ 하드웨어를 직접 제어하는 소프트웨어
→ 단순하고 빠르지만 복잡한 멀티태스킹은 어려움

RTOS
→ 실시간 작업을 관리하는 운영체제
→ 우선순위, task, scheduler, mutex, semaphore 제공

임베디드 리눅스
→ 복잡한 기능과 네트워크, 파일 시스템을 제공
→ 편리하지만 hard real-time 제어에는 부적합할 수 있음
```

## 31. 결론

펌웨어는 하드웨어 장치를 직접 제어하기 위해 장치 내부에 저장되는 소프트웨어이다.
하드웨어와 밀접하게 연결되어 있기 때문에 하드웨어와 소프트웨어의 중간 개념처럼 보이지만, 본질적으로는 하드웨어 제어를 위한 소프트웨어이다.

RTOS는 정해진 시간 안에 작업을 처리해야 하는 실시간 시스템을 위한 운영체제이다.
펌웨어가 단순한 제어 코드라면, RTOS는 여러 task의 우선순위와 실행 시간을 관리하는 시스템 소프트웨어이다.

우선순위는 중요한 작업을 먼저 실행하기 위해 필요하지만, 공유 자원을 잘못 사용하면 우선순위 역전이나 DeadLock이 발생할 수 있다.
이를 해결하기 위해 mutex, semaphore, priority inheritance, timeout, 자원 요청 순서 고정 같은 방법을 사용한다.

임베디드 시스템에서는 칩셋 성능, 메모리 용량, 실시간성 요구사항에 따라 펌웨어, RTOS, 임베디드 리눅스 중 적절한 소프트웨어 구조를 선택해야 한다.
